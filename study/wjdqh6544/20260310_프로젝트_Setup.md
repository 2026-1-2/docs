## Question
```
CCTV 원격 관제를 구현하고자 해.

카메라 -> 서버 -> 클라이언트 형태로 영상 정보가 전송되어야 하고,

클라이언트 -> 서버 -> 카메라 형태로 카메라를 조작하는 명령이 이동해야 함.

카메라 -> 서버는 RTSP 프로토콜을 이용하여 서버에 영상을 저장할건데, 합리적인 계획인가?
```

## Answer
네, **카메라에서 서버로 RTSP 프로토콜을 이용해 영상을 전송하고 저장하는 것은 매우 합리적이고 업계 표준에 가까운 계획입니다.** 대부분의 IP 카메라는 RTSP(Real-Time Streaming Protocol)를 기본적으로 지원하므로, 호환성이나 구현 난이도 측면에서 훌륭한 선택입니다.

성공적인 원격 관제 시스템 구축을 위해 현재 계획하신 아키텍처에서 고려해야 할 몇 가지 중요한 포인트와 추천 기술을 정리해 드립니다.

### 1. 영상 전송 및 저장 (Camera -> Server)

* **장점:** RTSP는 H.264, H.265와 같은 고효율 비디오 코덱을 지원하며, 지연 시간이 짧아 실시간 관제에 적합합니다.
* **저장 구현:** 서버에서 RTSP 스트림을 받아 저장하려면 **FFmpeg**나 **GStreamer** 같은 멀티미디어 프레임워크를 활용하는 것이 일반적입니다. 이를 통해 스트림을 디코딩 없이 직접 파일(예: `.mp4`, `.ts`)로 저장(Stream Copy)하면 서버의 CPU 부하를 크게 줄일 수 있습니다.

### 2. 클라이언트 영상 시청 (Server -> Client)

카메라에서 서버로는 RTSP가 좋지만, **서버에서 클라이언트(특히 웹 브라우저)로 영상을 보낼 때는 RTSP를 그대로 사용하기 어렵습니다.** 현대의 웹 브라우저는 RTSP를 기본적으로 지원하지 않기 때문입니다.

* **WebRTC:** 1초 미만의 초저지연이 필요한 실시간 관제에 가장 적합합니다. 서버에서 RTSP 스트림을 WebRTC로 변환하여 클라이언트에 제공해야 합니다.
* **HLS (HTTP Live Streaming):** 지연 시간(보통 3~10초)이 조금 있어도 괜찮다면 구현이 매우 쉽고 웹 브라우저 호환성이 가장 뛰어난 방식입니다.

### 3. 카메라 조작 명령 (Client -> Server -> Camera)

카메라의 렌즈 방향을 돌리거나 줌을 조절하는 PTZ(Pan-Tilt-Zoom) 제어를 위해서는 표준 프로토콜인 **ONVIF(Open Network Video Interface Forum)**를 사용하는 것을 권장합니다.

* **동작 방식:** 클라이언트가 서버로 제어 요청(HTTP REST API 또는 WebSocket)을 보내면, 서버는 카메라가 이해할 수 있는 ONVIF 규격(주로 SOAP 기반의 XML)으로 변환하여 카메라에 명령을 전달합니다.
* 거의 모든 RTSP 지원 IP 카메라는 ONVIF 프로토콜도 함께 지원하므로 하드웨어 종속성을 없앨 수 있습니다.

### 4. 보안 고려사항

RTSP 자체는 기본적으로 평문 전송이므로, 네트워크 중간에서 패킷 스니핑에 취약할 수 있습니다. 시스템이 퍼블릭 네트워크를 통과해야 한다면 다음을 고려해야 합니다.

* 카메라와 서버 간의 통신을 VPN이나 터널링으로 보호
* RTSPS(RTSP over TLS)를 지원하는 카메라 사용

## Question

나는 카메라 -> 서버로의 작업을 담당할 예정이라면, 카메라 -> 서버는 RTSP 를 써서 영상을 보낼건데.
카메라를 제어하기 위해, 서버에서 카메라로 명령을 어떻게 보내지?
```
## Answer
영상을 수신할 때 RTSP를 사용하더라도, 카메라의 방향을 돌리거나(PTZ) 설정을 변경하는 제어 명령은 보통 RTSP 스트림과는 **독립적인 별도의 통신 채널(Out-of-band)**을 통해 전송됩니다.

서버에서 카메라로 제어 명령을 보내는 가장 대표적인 방법은 크게 두 가지입니다.

### 1. ONVIF 프로토콜 사용 (업계 표준, 강력 추천)

가장 보편적이고 표준화된 방법입니다. ONVIF(Open Network Video Interface Forum)는 IP 카메라의 호환성을 위해 만들어진 글로벌 표준 프로토콜입니다.

* **통신 방식:** HTTP 기반의 SOAP(XML) API를 사용합니다. 서버에서 카메라의 ONVIF 포트(보통 80, 443, 또는 8080)로 XML 형태의 제어 명령을 HTTP POST 요청으로 보냅니다.
* **장점:** 제조사(한화, 하이크비전, 다후아 등)가 달라도 ONVIF Profile S 또는 Profile T를 지원하는 카메라라면 **동일한 코드**로 카메라를 제어할 수 있습니다.
* **구현:** 직접 XML을 파싱하고 생성할 필요 없이, 언어별로 잘 만들어진 라이브러리를 활용하면 됩니다.
* **Python:** `onvif-zeep`
* **Go:** `go-onvif`
* **Node.js:** `node-onvif`



### 2. 제조사별 HTTP/CGI API 사용 (Vendor-Specific API)

대부분의 IP 카메라는 내부에 소형 웹 서버를 내장하고 있으며, 자체적인 HTTP API(RESTful API 또는 CGI 스크립트)를 제공합니다.

* **통신 방식:** HTTP GET이나 POST 방식으로 카메라의 특정 URL에 요청을 보냅니다.
* *예시 (가상의 CGI API):* `http://[카메라IP]/cgi-bin/ptz.cgi?action=move&dir=left&speed=5`


* **장점:** ONVIF보다 직관적이고 가벼우며, 테스트하기 쉽습니다. (브라우저나 `curl` 명령어로 즉시 테스트 가능)
* **단점:** 카메라 제조사마다, 심지어 같은 제조사라도 모델 라인업에 따라 API 스펙이 달라서 제조사가 바뀌면 서버 쪽 코드를 다시 작성해야 합니다.

### 3. RTSP를 통한 인밴드(In-band) 제어 (제한적 사용)

RTSP 프로토콜 내에 `SET_PARAMETER`라는 메서드가 존재하여 이론적으로는 RTSP 커넥션 안에서 제어 명령을 보낼 수 있습니다.

* **한계:** 표준화가 덜 되어 있고 기능이 매우 제한적입니다. 최신 IP 카메라 환경에서는 PTZ 제어용으로 거의 사용하지 않으므로, 이 방식은 고려하지 않으셔도 무방합니다.

---

**요약하자면,**

* **영상 수신:** `Camera` --(RTSP/TCP 554)--> `Server`
* **명령 송신:** `Server` --(ONVIF/HTTP 80)--> `Camera`

위와 같이 두 개의 각기 다른 파이프라인(포트와 프로토콜)이 병렬로 돌아간다고 생각하시면 됩니다.

현재 도입 예정이신 카메라의 제조사가 특정되어 있나요? 원하신다면 서버 개발에 주로 사용하시는 언어(예: Python, Go 등)를 바탕으로 ONVIF PTZ 제어를 수행하는 간단한 코드 예시를 작성해 드릴 수 있습니다.

## Question)

IP카메라 제조사에서 독자적인 UI를 통해 실시간으로 영상을 확인하고 카메라를 조작할 수 있는 기능을 제공하는 상황에서, 내가 카메라를 조작하고 영상을 받아올 수 있나?

카메라 내부에서 UI 이외의 경로로부터의 접근을 차단하지는 않나?
```
## Answer)
제조사 전용 UI(웹페이지나 전용 앱)가 따로 있다고 해서 다른 접근을 꽉 막아두었을까 봐 걱정하시는 것은 시스템 연동을 계획할 때 매우 자연스럽고 중요한 포인트입니다.

결론부터 말씀드리면, **대부분의 상업용/기업용 IP 카메라는 제조사 전용 UI 외에도 사용자가 직접 영상을 받아오고 조작할 수 있도록 표준 통신 경로(RTSP, ONVIF 등)를 열어두고 있습니다.** 그 이유는 카메라 제조사들도 자사의 카메라가 다양한 NVR(네트워크 비디오 레코더)이나 써드파티 VMS(비디오 관리 시스템) 플랫폼과 연동되어 팔리기를 원하기 때문입니다. 폐쇄적으로 막아두면 B2B 시장에서 살아남기 어렵습니다.

제조사 전용 UI가 작동하는 방식과 사용자가 접근하는 방식을 비교해 보면 상황이 더 명확해집니다.

### 1. 제조사 UI도 결국 같은 방식으로 통신합니다

카메라 내부에 소형 리눅스 서버가 돌고 있다고 생각하시면 됩니다.

* 제조사에서 제공하는 웹 UI로 카메라 방향을 돌릴 때도, 결국 웹 브라우저가 카메라 내부의 웹 서버(HTTP 포트)로 API 요청을 보내는 것입니다.
* 전용 UI로 실시간 영상을 볼 때도, 내부적으로는 카메라의 스트리밍 서버(RTSP 포트 등)에서 영상을 끌어옵니다.
* 즉, **서버가 카메라에 보낼 요청(ONVIF, RTSP)과 전용 UI가 카메라에 보내는 요청은 본질적으로 같은 채널을 공유**합니다.

### 2. 단, 주의해야 할 3가지 예외 및 설정 사항

카메라가 접근을 아예 "차단"하지는 않지만, 보안이나 제품 타겟층에 따라 다음과 같은 허들이 있을 수 있습니다.

* **기능이 기본적으로 꺼져 있는 경우 (가장 흔함):**
최근 출시되는 카메라들은 보안 강화를 위해 출고 시점에 RTSP와 ONVIF 기능이 **비활성화(Disabled)** 되어 있는 경우가 많습니다. 이 경우 제조사 전용 UI에 최고 관리자로 로그인하여 '네트워크' 또는 '보안' 설정 메뉴에서 **"RTSP 활성화", "ONVIF 활성화"** 체크박스를 켜주어야 합니다.
* **전용 계정 생성 필요:**
ONVIF 통신을 하려면 별도의 ONVIF 전용 사용자(User)를 생성하고 권한(Administrator 또는 Operator)을 부여해야 하는 카메라(예: 하이크비전 등)도 있습니다.
* **가정용(클라우드 전용) 저가형 카메라인 경우 (주의):**
만약 샤오미, 티피링크(Tapo), 와이즈(Wyze) 같은 가정용 스마트 홈 카메라이고, 오직 '제조사 클라우드 앱'으로만 작동하도록 설계된 모델이라면 **내부 로컬 포트를 아예 막아두었을 수 있습니다.** 이런 모델들은 애초에 RTSP나 ONVIF를 지원하지 않거나, 편법(커스텀 펌웨어 등)을 써야만 가능하므로 원격 관제 시스템용으로는 부적합합니다.

---

**요약하자면,**
기업용/전문가용 IP 카메라(한화비전, 하이크비전, 다후아, 엑시스 등)라면 제조사 UI가 있더라도 서버에서 직접 RTSP로 영상을 당겨오고 ONVIF로 제어하는 데 아무런 문제가 없습니다. 단지 설정 화면에서 해당 기능을 켜주기만 하면 됩니다.

혹시 지금 사용을 고려 중이거나 보유하고 계신 카메라의 정확한 제조사와 모델명이 있으신가요? 모델명을 알려주시면 해당 기기가 RTSP와 ONVIF를 공식적으로 지원하는지, 연동 시 특별히 막혀있는 부분은 없는지 확인해 드릴 수 있습니다.

## Question
```
만약 카메라에서 별도의 RTSP / ONVIF 연결을 지원하지 않는다면 어떻게 해야 하지?
카메라 펌웨어를 해킹해야 하나?
```
## Answer
RTSP나 ONVIF 같은 표준 프로토콜을 전혀 지원하지 않는 폐쇄형 카메라(주로 샤오미, 헤이홈, Tapo 같은 가정용 IoT 카메라)를 사용해야만 하는 상황이라면, 펌웨어 해킹도 실제로 많이 쓰이는 방법 중 하나입니다.

하지만 목적과 상황에 따라 취할 수 있는 방법이 다릅니다. 이 경우 시도해 볼 수 있는 기술적 접근법들과 현실적인 조언을 정리해 드립니다.

### 1. 펌웨어 교체 (커스텀 펌웨어 설치)

질문하신 '펌웨어 해킹'에 해당하는 방법입니다. 기기 자체의 OS를 밀어버리고 RTSP/ONVIF를 지원하는 오픈소스 펌웨어로 덮어씌우는 방식입니다.

* **OpenIPC:** 가장 대표적인 IP 카메라용 오픈소스 리눅스 펌웨어 프로젝트입니다. 카메라 내부의 칩셋(SoC)이 지원 목록에 있다면, 기존 폐쇄형 펌웨어를 밀어내고 OpenIPC를 설치하여 완벽한 RTSP 스트리밍과 제어 권한을 얻을 수 있습니다.
* **기기별 해킹 펌웨어:** Wyze Cam, Xiaomi Dafang 등 유명 저가형 모델들은 전 세계 해커들이 만들어둔 전용 RTSP 펌웨어 패치(예: WyzeHacks, Dafang Hacks)가 존재합니다. SD 카드를 통해 손쉽게 패치할 수 있는 경우도 많습니다.

### 2. 네트워크 패킷 스니핑 (리버스 엔지니어링)

펌웨어를 건드리기 부담스럽다면, 카메라와 제조사 전용 앱(또는 웹 UI)이 통신하는 패킷을 가로채서 숨겨진 API나 스트리밍 주소를 찾아내는 방법이 있습니다.

* **Wireshark 활용:** 카메라와 스마트폰(또는 PC)을 같은 네트워크에 두고 중간에서 패킷을 캡처합니다.
* 전용 앱에서 카메라 방향을 돌릴 때 전송되는 HTTP 요청(URI, 파라미터, 헤더)을 분석하여, 서버에서 동일한 HTTP 요청을 쏘도록 스크립트를 작성합니다.
* **한계:** 통신이 암호화(HTTPS 등)되어 있거나 토큰 기반 인증을 빡빡하게 검사한다면 분석이 매우 까다롭습니다. 제조사가 앱을 업데이트하면서 API 주소를 바꿔버리면 시스템이 즉시 고장 납니다.

### 3. 비공식 API 라이브러리 활용

Tuya, Sonoff 같은 범용 IoT 플랫폼 기반의 카메라라면, 이미 전 세계 개발자들이 리버스 엔지니어링을 통해 만들어둔 비공식 Python/Node.js 라이브러리(예: `tuyapi`, `python-tuya`)가 존재할 확률이 높습니다. 이를 서버에 설치하여 제조사의 클라우드를 거쳐(우회하여) 카메라를 제어할 수 있습니다. 지연 시간이 길고 외부 인터넷 의존도가 높아진다는 단점이 있습니다.

### 현실적인 조언: 시스템의 목적이 무엇인가요?

기술적으로는 위와 같은 우회 방법들이 존재하지만, **안정적인 원격 관제 시스템 구축이 목적이라면 해당 카메라의 사용을 포기하고 표준(RTSP/ONVIF)을 지원하는 카메라로 교체하는 것을 강력히 권장합니다.**

* **유지보수의 지옥:** 리버스 엔지니어링이나 해킹 펌웨어에 의존하는 시스템은 카메라 제조사의 사소한 업데이트나 클라우드 정책 변경 한 번에 전체 관제 시스템이 마비될 수 있습니다.
* **보안 취약점:** 임의로 변조된 펌웨어나 검증되지 않은 백도어 API를 사용하는 것은 관제 시스템의 근본적인 보안을 위협합니다.

현재 구상 중이신 원격 관제 시스템이 개인적인 학습이나 취미용 토이 프로젝트이신가요, 아니면 실제 서비스나 비즈니스 현장에 도입하기 위한 상용 프로젝트이신가요?
